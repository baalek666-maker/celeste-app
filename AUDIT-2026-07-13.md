## 🔴 BUGS CRITIQUES (P0 — impact immédiat)

### 1. **Le bouton "Se déconnecter" ne fonctionne pas**
**Fichier :** `src/screens/Settings.tsx:210` + `src/lib/storage.ts:76`

```ts
// storage.ts logout() — supprime celeste_user, celeste_journal, etc.
// MAIS NE SUPPRIME PAS le JWT token 'celeste_jwt'
```

`handleLogout` appelle `logout()` qui clear `celeste_user` mais **pas le JWT** (`celeste_jwt`). Après `window.location.reload()`, `App.tsx` appelle `getToken()` → token toujours là → `api.getProfile()` → **l'utilisateur reste connecté**.

**Fix :** Ajouter `localStorage.removeItem('celeste_jwt')` dans `logout()`, ou importer `clearToken` depuis `api.ts` et l'appeler.

---

### 2. **Cache horoscope weekly cassé → LLM appelé à chaque fois**
**Fichier :** `server/server.js:1413`

```js
const cached = db.prepare('SELECT summary FROM horoscope_cache WHERE user_id = ? AND date = ?').get(...);
//                                          ^^^^^^^^
// La table horoscope_cache a une colonne 'content', PAS 'summary'
```

La colonne `summary` n'existe pas → `cached?.summary` retourne toujours `undefined` → le cache weekly ne hit jamais → **régénération LLM systématique** (coût + latence énormes pour le weekly view).

**Fix :** `SELECT content FROM horoscope_cache ...` puis adapter la déstructure.

---

### 3. **Scan compatibilité décrémenté AVANT le LLM → scan perdu sur échec**
**Fichier :** `server/server.js:1641` vs `1673`

```js
// Ligne 1641 : décrément AVANT l'appel LLM
db.prepare('UPDATE users SET scans_remaining = scans_remaining - 1 WHERE id = ?').run(req.user.id);
// ... plus tard, ligne 1673 :
} catch (llmError) {
  // Le scan est déjà consommé, mais l'utilisateur n'a aucun résultat
  return res.status(500).json({ error: 'Compatibility generation failed' });
}
```

Contrairement à la route horoscope qui a un `FALLBACK_HOROSCOPES`, la route compatibilité **n'a pas de fallback** → en cas d'échec LLM, l'utilisateur perd un scan sans rien obtenir.

**Fix :** Déplacer le décrément après le succès du LLM, ou implémenter un remboursement dans le catch.

---

### 4. **Mots de passe en clair dans les logs serveur**
**Fichier :** `server/server.js:681` (middleware logging)

```js
console.log(`[POST IN] ${req.path}`, JSON.stringify({ body: req.body, ip, ua }));
```

Ce middleware loggue **tous les POST bodies**, y compris `password` pour `/api/auth/login` et `/api/auth/register`. Faille de sécurité majeure — les mots de passe sont persistés en clair dans les logs serveur.

**Fix :** Filtrer les champs sensibles :
```js
const safeBody = { ...req.body };
delete safeBody.password;
console.log(`[POST IN] ${req.path}`, JSON.stringify({ body: safeBody, ip, ua }));
```

---

### 5. **`JSON.parse(r.birth_data)` sans guard null → crash 500**
**Fichier :** `server/server.js:1282` (route GET /api/profiles/:id)

```js
birthData: JSON.parse(r.birth_data),  // crash si r.birth_data est NULL
```

Si un profil a été créé sans `birth_data` (possible via certains chemins), cette ligne fait planter la route avec une 500.

**Fix :** `birthData: r.birth_data ? JSON.parse(r.birth_data) : null`

## 🟠 BUGS MAJEURS (P1 — impact fort)

### 6. **Paywall perd l'objet User lors de la restauration un abonnement**
**Fichier :** `src/screens/Paywall.tsx:57-61`

```tsx
onSubscribe({
  email: '', birthData: null, natalChart: null,  // ← écrase l'user existant
  ...
})
```

Le callback `onSubscribe` reconstruit un User minimal qui **perd l'email, birthData, natalChart**. Si le parent met à jour son state avec cet objet, l'utilisateur perd ses données.

**Fix :** Fusionner avec l'User existant : `onSubscribe({ ...existingUser, isPremium: true, ... })`

---

### 7. **Offline queue non vidée au logout → actions cross-user**
**Fichier :** `src/lib/storage.ts:76` + `src/lib/offlineQueue.ts`

`logout()` ne supprime pas `celeste:offline_queue:v1`. Les mutations en file d'attente (journal entries, etc.) seront rejouées au prochain login, potentiellement pour un **autre utilisateur**.

**Fix :** Ajouter `localStorage.removeItem('celeste:offline_queue:v1')` dans `logout()`.

---

### 8. **Paywall onClose renvoie toujours à 'home' (perte de contexte)**
**Fichier :** `src/App.tsx:366`

```tsx
<Paywall onClose={() => setScreen('home')} ... />
```

Si le paywall est déclenché depuis Horoscope ou Compatibilité, fermer le paywall renvoie à Home au lieu de revenir à l'écran précédent.

**Fix :** Mémoriser l'écran précédent avant d'ouvrir le paywall.

---

### 9. **`scans_remaining` peut devenir négatif**
**Fichier :** `server/server.js:1671`

```js
scansRemaining: isPremium ? null : (user.scans_remaining - 1)
// Si user.scans_remaining est déjà 0 → retourne -1
```

La route compatibilité décrémente sans vérifier le floor à 0 dans la réponse.

**Fix :** `Math.max(0, user.scans_remaining - 1)`.

---

### 10. **Log backend hardcode "45s" même si timeoutMs varie**
**Fichier :** `server/server.js:~410`

```js
console.log(`[LLM] attempt ${attempt} timed out after 45s`);
//                                                    ^^^^ hardcoded
```

Le timeout est paramétrable (120000ms pour portraits), mais le log hardcode toujours "45s" → debugging trompeur.

## 🟡 BUGS MODÉRÉS (P2 — impact moyen)

### 11. **Aucun test automatisé**
Aucun fichier `.test.ts` / `.spec.ts` dans le projet. La logique de gamification, cache, LLM fallback, et navigation n'est pas testée. Risque élevé de régression silencieuse.

### 12. **Chunk JS à 543 KB (gzip 161 KB)**
Le bundle principal dépasse 500 KB. Aucun code-splitting. Conséquence : temps de chargement initial + lenteur sur mobile 3G/4G.

**Fix :** `React.lazy()` sur les screens (Horoscope, Compatibility, Tarot, AstroPortrait).

---

### 13. **`addJournalEntry` fait un no-op write inutile**
**Fichier :** `src/lib/storage.ts:99`

```ts
saveUser({ ...getUser(), natalChart: getUser().natalChart }); // no-op write
```

Appelle `getUser()` 3 fois + écrit dans localStorage pour rien.

---

### 14. **VULNÉRABILITÉ : route `app.delete('/api/account')` sans soft-delete**
La suppression de compte est irréversible. En cas de misclick sur "Supprimer mon compte" (Settings), toutes les données sont perdues (11 tables en cascade).

**Recommandation :** Soft-delete avec fenêtre de grâce 30 jours.

---

### 15. **Service Worker ne cache pas les index de tarot**
Le SW cache les assets statiques mais `getTarotImage()` charge des images dynamiques non pré-cachées → sur mobile, le tirage peut afficher un fallback emoji si les images ne sont pas encore chargées.

---

### 16. **Streak logic peut être trompé via timezone**
**Fichier :** `server/gamification.js`

Le streak est basé sur `new Date().toISOString()` (UTC). Un utilisateur peut prolonger son streak en changeant de timezone. Mineur car l'impact est faible.
